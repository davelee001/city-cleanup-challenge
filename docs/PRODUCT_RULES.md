# City Cleanup Reward Product Rules

This document defines the first buildable version of the City Cleanup reward
workflow. It is intentionally conservative: cleanup evidence and verification
remain off-chain, while Celo is used only for wallet ownership and approved
reward payments.

## 1. MVP Goal

A registered user submits evidence of solid waste they cleaned. The platform
checks that the evidence is authentic, unique, and eligible. A reviewer approves
uncertain submissions. Each approved submission can trigger exactly one CELO
payment to the user's verified wallet.

## 2. User Eligibility

- A user must have a registered account.
- A user must verify control of a Celo wallet by signing a one-time message.
- One wallet may be linked to only one active account unless an administrator
  documents an approved exception.
- Suspended accounts and wallets cannot submit evidence or receive rewards.
- The application never requests or stores a wallet seed phrase or private key.

## 3. Accepted Cleanup Evidence

The MVP accepts bottles and other visible solid waste, including plastic, glass,
metal cans, paper/cardboard, and mixed non-hazardous litter.

Each submission requires:

- One in-app photo before cleanup.
- One in-app photo after cleanup showing the collected waste.
- Capture timestamps generated by the application.
- GPS coordinates and location accuracy, collected with user consent.
- A waste category.
- An item count or estimated weight.
- An optional note describing the cleanup.

Hazardous, medical, biological, or unidentifiable waste is not reward-eligible
in the MVP and should be handled only under an approved safety program.

## 4. Verification Rules

Every submission receives:

- A cryptographic file hash for exact duplicate detection.
- A perceptual image hash for resized, cropped, or compressed duplicates.
- A comparison against images from the same user and all previous submissions.
- Timestamp, GPS, and metadata consistency checks.
- A before/after scene-consistency check.
- Synthetic-image and manipulation risk scores.
- Account, device, submission-frequency, and wallet risk checks where legally
  permitted.

AI-generated-image detection is a risk signal, not proof. The system must combine
multiple signals and send uncertain cases to a human reviewer.

Exact duplicates and previously paid evidence are automatically rejected.
High-confidence valid submissions may be automatically approved only after the
verification thresholds have been tested with real pilot data.

## 5. Submission Lifecycle

The allowed state progression is:

`draft -> submitted -> automated_review -> manual_review -> approved -> payment_pending -> paid`

A submission may move from either review state to `rejected`. A rejected
submission may move to `appealed`, then return to `manual_review`.

Every transition records the actor, timestamp, previous state, new state, and
reason. An approved or paid submission cannot return to an editable state.

## 6. Review and Appeals

- Reviewers see the before/after evidence, duplicate matches, location evidence,
  verification scores, and the user's submission history.
- A reviewer cannot approve their own submission.
- Rejections require a reason visible to the user.
- A user may appeal once within a configurable period.
- A different reviewer should decide an appeal whenever staffing permits.

## 7. CELO Reward Rules

- The MVP pays native CELO, not a custom token.
- Testnet CELO is used until the complete workflow passes the pilot phase.
- The production reward amount remains configurable and must not be hard-coded
  in the mobile application or smart contract client.
- The reward engine records the rule version and calculation used for each
  submission.
- Per-submission, daily, weekly, account, and wallet payout caps are required.
- Large or unusual rewards require manual approval.
- Each payment uses the immutable submission ID as its unique claim ID.
- The database and smart contract both reject a claim ID that was already paid.
- A transaction is marked `paid` only after the required on-chain confirmations.

Initial production reward values, payout caps, and treasury size require a
separate financial decision after testnet measurements establish expected
submission volume, fraud rate, and transaction cost.

## 8. Privacy and Data Handling

- Photos, GPS coordinates, identity data, and verification results stay
  off-chain.
- Only the claim ID, wallet, amount, and payment event need to be recorded
  on-chain.
- Images are stored privately and accessed through expiring URLs.
- The user must consent before location collection.
- Retention and deletion periods must be documented before a public launch.

## 9. MVP Success Criteria

The MVP is ready for a controlled testnet pilot when:

- A user can register and verify a Celo wallet.
- A user can submit before/after evidence from the application.
- Exact and perceptual duplicates are detected.
- A reviewer can approve or reject uncertain evidence.
- An approved submission creates one idempotent testnet CELO payment.
- Retrying a payment cannot pay the same submission twice.
- The user can see the verification result and transaction hash.
- Administrators can pause rewards and audit every decision.

## 10. Decisions Required Before Mainnet

- Final CELO reward per approved submission.
- Quantity or waste-type multipliers.
- Daily and weekly payout caps.
- Eligible geographic areas.
- Minimum age and identity requirements.
- Evidence and account retention periods.
- Appeal response time.
- Treasury ownership and multisig signers.
- Required blockchain confirmation count.
- Legal terms, privacy policy, and prohibited-waste policy.
